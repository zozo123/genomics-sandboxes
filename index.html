01

What I was actually measuring

The plan was boring on purpose: boot one machine, warm it once, snapshot it, and fork a copy per chromosome — just to see how much faster a frozen reference makes the next run. No biology goals. An infrastructure stopwatch. Three reasons it was worth doing at all:

  1. You can just read it yourself. A 23andMe export, a Nebula or Nucleus whole-genome file, the public reference — it's all just text: four letters, a few billion times. The barrier was never the data. It was the machine.
  2. The interesting signal is regulatory, not spelling. Most consumer genomics stops at variants. But what your cells do with the same DNA is set by methylation at 3,201,954 CpG sites — the switches behind biological-age clocks, tumor screens, and cell identity. That's what the kernel counts.
  3. A sandbox is a computer that dies on purpose. In March 2025, 23andMe's Chapter 11 put ~15 million people's DNA on the auction block; twenty-seven state attorneys general sued. "Analyze your genome on a computer that dies on purpose" stopped sounding paranoid. The box boots, reads, reports, and vanishes — your data never outlives the question.

the moveBring your own genome. Shard it, fork a warm reference once, scan, reduce — then throw the computers away.

02

The process: warm → snapshot → fork → reduce

Here is the entire experiment, in the order it happened. No step is clever. The one idea is that the reference genome is read-only — so you should only ever pay to load it once. MapReduce's real trick was never "go parallel"; it was broadcasting that read-only base once. A VM snapshot is that broadcast.

  1. Warm — build the base once 24.1sBoot one machine and let it do the expensive, boring part exactly once: pull the GRCh38 chromosomes, build the index, load the libraries, settle the page cache. Twenty-four seconds nothing downstream should ever repeat — the moment before the broadcast.
  2. Snapshot — freeze the warm machine 127 MB · 3.9sFreeze the whole warmed machine — disk, caches, the loaded reference — to a 127 MB image in 3.9s. This frozen image is the broadcast: instead of shipping the reference to every worker, you hand each one the machine that already has it. Off-the-shelf (CRIU, Firecracker copy-on-write, Lambda SnapStart) — pointed at a genome instead of a web app.
  3. Fork → map — one copy per chromosome 5 shards · 14.8sFork five machines from that one snapshot — chr19, 20, 21, 22, Y — each a copy-on-write child that starts already warm. Every fork is a mapper: scan its chromosome, count CpG sites, call CpG-island candidates. Warm, the five finish in 14.8s wall-clock; cold — each re-staging the reference itself — the same fan-out takes 34.2s. Same answer; you just paid for the reference once instead of five times.
  4. Reduce — merge into one genome → 14.8s rerunCollect five per-chromosome reports into one genome-wide picture — 277,817,649 bp, 3,201,954 CpG sites, 37,374 island candidates — then destroy every box. First warm run end-to-end: 42.8s. Every rerun after, reusing the snapshot: 14.8s.

why a snapshot and not a container imageA container image is the cold start: every worker pulls and re-provisions. A VM snapshot is the warm start — disk and process state captured at an instant, restored copy-on-write. Skip the boot, skip the staging, fork the finished state.

the control plane is localThere is no cluster to manage. The entire master and reducer is one ~80-line shell script on a laptop — scripts/run_demo.sh — driving the islo CLI: fork, collect each shard's JSON, merge, delete. This whole run was orchestrated exactly that way by an agent (Claude Code) calling the same four verbs. The controller is disposable too; only the snapshot persists.

the turnThat was the whole point of the exercise — measure the speedup. Then I looked at the per-chromosome counts the mappers returned, and the benchmark stopped being about speed.

03

The run

Five human chromosomes from GRCh38 — chr19, chr20, chr21, chr22, chrY — 278 Mb in all. Each shard computed GC% and CpG observed/expected in 1 Mb bins, and counted CpG-island candidates (Gardiner-Garden & Frommer: 200 bp window, GC > 50%, obs/exp > 0.6).

04

What the snapshot actually buys

The naive way: every worker stages the reference itself. The snapshot way: stage once, fork. The first run is faster; the real payoff is every run after — and the redundant work you never do.

the honest versionFirst run: a modest 2.3×, and I'll say so. The real win is the second question you ask — 7.4× on rerun, because the broadcast is already frozen. Swap the 72 MB of test chromosomes for a real 3 GB reference plus BWA indices, fan out across a whole cohort, and the snapshot stops being an optimization and becomes the only sane way to do it. Every fork also starts byte-identical — which is reproducibility, the thing genomics actually grades you on.

05

Reproduce it

The whole demo is four verbs of the islo CLI. No Kubernetes, no Nextflow, no cluster account.

# 1 · warm base: toolchain + GRCh38 chromosomes + index (paid once)
islo use gx-warm -- bash -lc './warmup.sh chr19 chr20 chr21 chr22 chrY'

# 2 · broadcast: freeze the warm box to a snapshot
islo snapshot save gx-warm --name genomics-warm

# 3 · MAP: fork one warm box per chromosome, in parallel
for chr in chr19 chr20 chr21 chr22 chrY; do
  islo use gx-map-$chr --snapshot genomics-warm -- \
    python3 compute.py $chr &        # each starts warm, just computes
done; wait

# 4 · REDUCE: merge per-shard JSON → genome-wide landscape, then delete the boxes
python3 reduce.py *.json > landscape.json

⌗

Objections

"This is GC content, not medicine." Correct — and on purpose. It's a transparent, verifiable proxy that exercises the exact pattern real pipelines use (scatter by interval, gather results). Swap the kernel for DeepVariant or a methylation caller; the map-reduce-on-snapshots shape doesn't change.

"Your island calls are loose." They are candidates. The Gardiner-Garden & Frommer 1987 rule (200 bp, GC > 50%, obs/exp > 0.6) over-calls GC-rich repeats; Takai & Jones (2002) tightened it to 500 bp / GC > 55% / obs/exp > 0.65. The point here is the shape and gradient across chromosomes, which both rules agree on — not a curated annotation like ENCODE's cCRE Registry (Moore et al., Nature 2020).

"Chromosomes aren't balanced shards." True — chrY is mostly gaps, chr20 is the biggest here. The warm fork wall-clock is set by the slowest shard, which is why real pipelines shard by fixed intervals. The snapshot argument is orthogonal to how you cut the work.

"Is forking a snapshot really reproducible?" More than re-provisioning is. Every shard restores the same captured disk state, copy-on-write — not "the same Dockerfile, fingers crossed the mirrors agree today."

"Privacy theater?" The box is ephemeral and isolated, and it's yours for its lifetime. Your genome is analyzed in a computer you control and then destroy — not deposited into a third party's permanent, sellable database.
