Copy-on-write fan-out for genomic scatter-gather, validated against a positive control.

Yossi Eliaz · technical report · 2026 · run on islo.dev

Abstract Genomic scatter-gather distributes a large, read-only reference — a FASTA and its derived indices, 8–40 GB in practice — to every shard of a fan-out, re-localizing it per task and re-staging it after every preemption. We evaluate an alternative: warm one worker until the reference is resident, snapshot its initialized state once, and realize each shard as a restore-and-fork of that snapshot, so the reference is shared copy-on-write. Run genome-wide on islo.dev — all 24 GRCh38 chromosomes, 3.09 Gb — the warm base builds once in 88.6 s, snapshots to 962 MB in 10.3 s, and a 24-way fan-out then completes in ~60 s of wall-clock; staging the reference per worker instead would cost ≈13 min serially (13× on snapshot reuse). The whole pipeline — warm, snapshot, fork, reduce — is driven end-to-end by a Claude Code agent acting as the harness. Correctness is checked with a positive control: per-chromosome CpG-island density, an established correlate of gene density, is recovered in the expected order (chr19 densest, chr4 sparsest). We discuss the implication for automated, model-written analysis and state the limits.

01

Background

Sequencing cost has fallen faster than compute: a 30× human genome is now on the order of $100 [9]. The binding constraint has moved downstream to analysis, and within analysis to an unglamorous cost — moving the reference. Three observations motivate the method.

  1. The expensive state is read-only and shared. The reference FASTA (~3 GB) and its derived indices (BWA-MEM ~5 GB; a STAR index 27–30 GB and ~30 GB resident; GATK known-sites bundles) total 8–40 GB of immutable bytes. Standard scatter-gather workflows [6][7] fan a single run across 50–1000 genomic intervals and re-localize that identical reference to every shard, re-staging it after each spot/preemptible interruption.
  2. Reproducibility is a first-class requirement. Genomics grades results on reproducibility, yet re-staging "the same" reference per worker depends on mirrors and tool versions agreeing at run time. Inputs that are identical by construction are a stronger guarantee than inputs that are identical by convention.
  3. The kernel is increasingly model-written. When the per-shard analysis is generated by a model rather than reviewed by a person, untrusted code requires disposable isolation, and its output requires a machine-checkable control before it is trusted.

measured quantity · CpG-island densityThe kernel computes one quantity per chromosome. A CpG island is a window in which the dinucleotide CpG occurs far above its genome-wide expectation [3][4]; such islands coincide with a large fraction of human gene promoters, which makes their density a coarse but well-characterized correlate of gene density [1]. The kernel reads candidate islands off the reference sequence (29,401,360 CpG sites genome-wide); it does not measure methylation, which is cell-type-specific and absent from the reference.

02

Methods

In a real genomics pipeline the expensive state is read-only and shared: the reference FASTA and its derived indices. A GRCh38 FASTA is about 3 GB; a BWA-MEM index adds ~5 GB, a STAR index 27–30 GB (and wants ~30 GB of RAM resident), plus GATK known-sites bundles — 8–40 GB of immutable bytes that never change during a run. Best-practice workflows (nf-core, GATK Best Practices) scatter-gather over hundreds of intervals — a HaplotypeCaller run is routinely fanned across 50–1000 shards — and the conventional answer re-localizes that identical reference to every task — re-downloading multi-gigabyte indices per shard, and re-staging them again on every spot/preemptible interruption.

Snapshot-fork inverts it. Warm one worker — open the reference, load the indices, page them resident — then snapshot that initialized address space once. Each shard is a restore-and-fork from that snapshot — the mechanism Firecracker microVMs [5] and AWS Lambda's on-demand loading [8] already use: restore maps the saved memory MAP_PRIVATE, so the reference and index pages are shared copy-on-write and only the pages a fork actually writes diverge. The analysis never writes the index, so it stays shared. Broadcasting the reference to the (N+1)th shard costs a page-table setup, not another multi-gigabyte read — and it's byte-identical by construction, because every fork maps the same physical pages instead of doing N independent loads that merely ought to agree.

  1. Warm — build the base once 88.6sBoot one machine and let it do the expensive, boring part exactly once: pull all 24 GRCh38 chromosomes, build the index, load the libraries, settle the page cache. About 90 seconds of work that nothing downstream should repeat — the step before the broadcast.
  2. Snapshot — freeze the warm machine 127 MB · 3.9sFreeze the whole warmed machine — disk, caches, the loaded reference — to a 127 MB image in 3.9s. This frozen image is the broadcast: instead of shipping the reference to every worker, you hand each one the machine that already has it. Off-the-shelf (CRIU, Firecracker copy-on-write, Lambda SnapStart) — pointed at a genome instead of a web app.
  3. Fork → map — one box per chromosome 24 shards · ~60sFork one machine per chromosome (chr1…chr22, X, Y) from that one snapshot — each a copy-on-write child that starts already warm — in waves of eight. Every fork is a mapper: scan its chromosome, count CpG sites, call CpG-island candidates. Warm, the whole genome-wide fan-out finishes in ~60 s of wall-clock; cold, with each worker re-staging the reference itself, the same work would run ≈13 min serially. The result is identical; the reference is paid for once, not 24 times.
  4. Reduce — merge into one genome → rerun ≈ fan-outCollect the per-chromosome reports into one genome-wide picture — 3,088,269,832 bp, 29,401,360 CpG sites, 264,816 island candidates — then destroy every box. First end-to-end run (warm + snapshot + fan-out): ~159 s. Every rerun after, reusing the snapshot: only the fan-out (~60 s).

MechanismA container image is a cold start: every worker pulls and re-provisions. A VM snapshot is a warm start — disk and process state captured at an instant and restored copy-on-write — so the boot and the staging are skipped and the finished state is forked instead [5][8].

Orchestration — the harness is the agentThere is no scheduler and no cluster. The control plane is a Claude Code agent: it runs crabbox.sh / the islo CLI, warms the box, snapshots it, forks one shard per chromosome in waves, collects each shard's JSON, reduces, and tears the boxes down. In this genome-wide run a Claude Code sub-agent drove the entire 24-way fan-out end-to-end and reported the receipts the figures above are built from. You run it the same way: tell the agent to do it. The controller is itself disposable; only the snapshot persists.

The validation quantity is candidate CpG islands per megabase of sequenced chromosome — the raw per-shard output, sorted descending (Figure 3).

Positive controlCpG-island density is an established correlate of gene density [1][3][4], so a correctly wired scatter-gather must reproduce that gradient. Across all 24 chromosomes the ranking comes out as expected — chr19 densest (287.6 islands per sequenced Mb; chr19 carries ~25 protein-coding genes/Mb, roughly double the genome average [2]) down to chr4 sparsest (62.4, a gene-poor chromosome). Recovering a known gradient from an independently written kernel is evidence that the sharding, the per-shard computation, and the reduce step are jointly correct: a dropped shard, a coordinate off-by-one, or double-counted bins would distort it.

This is the whole genome, not a sample of it. The ranking tracks gene density broadly, with the coarse-metric wobble you'd expect — a few mid-density chromosomes swap rank, and counts are per sequenced (non-N) megabase so gene-poor heterochromatin (chrY, the acrocentric short arms) doesn't deflate its neighbours. Recovering the gradient genome-wide is therefore consistent with a correct pipeline rather than proof of one: a positive control, not a unit test.

ScopeOnce the snapshot is frozen, every rerun pays only the fan-out — about 13× faster than staging the reference on each worker, and 862 MB of reference transfer that never happens again. This warm base is the ~900 MB gzipped reference — still smaller than the 3 GB FASTA plus 5–30 GB resident indices a real aligner holds, so the demonstration establishes the mechanism at genome scale, not the full at-scale economics. A snapshot also restores cleanly only on a compatible CPU and kernel; copy-on-write sharing holds only for pages nothing writes; and byte-identical output additionally requires pinning clocks, RNG seeds, and scratch state.

05

Application: automated, model-written analysis

The method matters most when the per-shard analysis is generated and executed by software rather than written and reviewed by a person. Three properties make that case.

  1. Untrusted model code gets disposable isolation. A model-written pipeline is code you didn't review, running against your reference data. Each shard is a fork off a clean, known-good warmed image; it runs and is discarded. A buggy or adversarial pipeline can only corrupt its own copy-on-write pages — the shared read-only reference is never writable, so it can't be mutated across shards.
  2. Reference broadcast is O(1) and byte-identical. Agents drive fan-out into the thousands — parameter sweeps, per-sample and per-interval reruns, per-tool variants — where the only shared thing is the expensive 8–40 GB reference state. Copy-on-write forking makes the marginal cost of the thousandth shard a page-table setup, not a thousandth multi-gigabyte load — and every fork provably starts from the identical physical pages, not N loads that merely should agree.
  3. Positive controls make model output trustworthy. Model output is only trustworthy if it's reproducible and checked. Baking a known-answer control into the warm-up — here, the CpG-island/gene-density gradient — gives each fork a ground-truth check it must reproduce before its real result is trusted. Byte-identical input pages plus a positive control turn "the model ran something" into "the model ran something that passed a check."

SummaryDisposable isolation, an O(1) byte-identical broadcast, and a baked-in positive control are exactly the guarantees an automated pipeline needs: an analysis environment that is instantiated, used, checked, and discarded rather than maintained.

06

Reproducibility and availability

The whole genome-wide run is one harness script, crabbox.sh, driven by a Claude Code agent on the islo fabric — no scheduler, no cluster account. Re-forking the retained snapshot reproduces a per-chromosome result byte-for-byte (verified on chr21).

# the agent runs the harness end-to-end: warm → snapshot → fork(24) → reduce → teardown
./crabbox.sh run

# which is, in islo verbs:
# 1 · warm base — all 24 chromosomes + index, paid once
islo use gx-warm -- bash -lc './warmup.sh chr1 chr2 ... chr22 chrX chrY'
# 2 · broadcast — freeze the warm box to a snapshot
islo snapshot save gx-warm --name genomics-wg
# 3 · MAP — fork one warm box per chromosome (waves of 8)
for chr in chr1 ... chrY; do
  islo use gx-$chr --snapshot genomics-wg -- python3 compute.py $chr &
done; wait
# 4 · REDUCE — merge per-shard JSON → genome-wide receipts, then delete the boxes
python3 scripts/reduce_wg.py ...

07

Limitations and objections

"This is GC content, not medicine." Correct — and on purpose. It's a transparent, verifiable proxy that exercises the exact pattern real pipelines use (scatter by interval, gather results). Swap the kernel for DeepVariant or a methylation caller; the map-reduce-on-snapshots shape doesn't change.

"Your island calls are loose." They are candidates. The Gardiner-Garden & Frommer 1987 rule (200 bp, GC > 50%, obs/exp > 0.6) over-calls GC-rich repeats; Takai & Jones (2002) tightened it to 500 bp / GC > 55% / obs/exp > 0.65. The point here is the shape and gradient across chromosomes, which both rules agree on — not a curated annotation like ENCODE's cCRE Registry (Moore et al., Nature 2020).

"Chromosomes aren't balanced shards." True — chr1 (~250 Mb) is ~5× chr21, so the fan-out wall-clock is set by the largest shard, which is why production pipelines shard by fixed intervals instead. The snapshot/broadcast argument is orthogonal to how you cut the work; here we shard by chromosome only to make the positive control legible.

"Is forking a snapshot really reproducible?" More than re-provisioning is. Every shard restores the same captured disk state, copy-on-write — not "the same Dockerfile, fingers crossed the mirrors agree today."

"Privacy theater?" The box is ephemeral and isolated, and it's yours for its lifetime. Your genome is analyzed in a computer you control and then destroy — not deposited into a third party's permanent, sellable database.
